```text

恢复技术：数据库坏了、事务失败了，怎么恢复到正确状态
并发控制：多个事务同时执行，怎么避免互相干扰
```

概念题常见问法：

```text
1. 什么是三级模式结构？优点是什么？

9. 并发操作会带来哪些数据不一致问题？
10. 什么是封锁？S 锁、X 锁是什么？
11. 什么是三级封锁协议、两段锁协议？
12. 死锁如何检测和解除？
```

# 1. 第一章 绪论

## 1.1 数据、数据库、数据库管理系统、数据库系统

### 1.1.1 数据 Data

**数据**是描述事物的符号记录。

例如：

```text
姓名：张三
年龄：20
成绩：88
商品号：G01
```

数据不只是数字，也可以是文字、图像、音频、视频等。

考试答法：

```text
数据是描述事物的符号记录，是数据库中存储和管理的基本对象。
```

---

### 1.1.2 数据库 DB

**数据库**是长期存储在计算机内、有组织的、可共享的大量数据集合。

关键词：

```text
长期存储
有组织
可共享
数据集合
```

考试答法：

```text
数据库是长期存储在计算机内、有组织、可共享的大量数据集合。数据库中的数据按照一定的数据模型组织、描述和存储，具有较小冗余度、较高数据独立性和易扩展性。
```

---

### 1.1.3 数据库管理系统 DBMS

**DBMS 是位于用户和操作系统之间的一层数据管理软件。**

这是选择题高频点。

```text
用户 / 应用程序
     ↓
DBMS
     ↓
操作系统
     ↓
数据库文件
```

DBMS 的主要功能：

```text
1. 数据定义功能：定义数据库、表、视图、索引等
2. 数据操纵功能：查询、插入、删除、修改
3. 数据库运行管理：并发控制、安全性控制、完整性控制、恢复
4. 数据组织、存储和管理
5. 数据库建立和维护
6. 通信接口
```

选择题注意：

```text
DBMS 介于：用户 与 操作系统 之间
```

不要被“数据库和操作系统之间”“用户和应用程序之间”绕进去。

---

### 1.1.4 数据库系统 DBS

**数据库系统**通常由以下部分组成：

```text
数据库 DB
数据库管理系统 DBMS
应用程序
数据库管理员 DBA
用户
硬件、软件环境
```

考试答法：

```text
数据库系统是由数据库、数据库管理系统、应用程序、数据库管理员和用户等组成的系统。
```

---

## 1.2 数据管理技术的发展阶段

数据管理技术一般经历三个阶段：

```text
人工管理阶段 → 文件系统阶段 → 数据库系统阶段
```

### 1.2.1 人工管理阶段

特点：

```text
数据不保存或保存很少
应用程序自己管理数据
数据不共享
数据冗余大
数据不独立
```

### 1.2.2 文件系统阶段

特点：

```text
数据可以长期保存
由文件系统管理数据
数据共享性差
数据冗余较大
数据独立性差
应用程序和数据结构联系紧密
```

### 1.2.3 数据库系统阶段

特点：

```text
数据结构化
数据共享性高
数据冗余小
数据独立性高
由 DBMS 统一管理和控制
数据一致性高
安全性、完整性、并发控制和恢复能力强
```

选择题常考：

```text
相对于文件系统，关系数据库/数据库系统不具有“数据冗余高”的特点。
数据库系统阶段数据独立性最高。
```

---

## 1.3 数据库系统的特点

常考答案：

```text
1. 数据结构化
2. 数据共享性高、冗余度低
3. 数据独立性高
4. 数据由 DBMS 统一管理和控制
```

### 1.3.1 数据结构化

数据库中的数据不是孤立存放，而是按照数据模型组织起来。

例如学生、课程、选课之间有联系。

---

### 1.3.2 数据共享性高，冗余度低

多个用户、多个应用程序可以共享同一份数据。

共享性高可以减少重复存储，所以冗余度低。

---

### 1.3.3 数据独立性高

**数据独立性**是指应用程序和数据结构之间相互独立，数据结构变化时，应用程序尽量少改或不改。

分为：

```text
逻辑独立性
物理独立性
```

#### 逻辑独立性

含义：

```text
数据库的全局逻辑结构改变时，外模式和应用程序可以尽量不变。
```

由：

```text
外模式 / 模式映像
```

保证。

#### 物理独立性

含义：

```text
数据库的物理存储结构改变时，模式和应用程序可以尽量不变。
```

由：

```text
模式 / 内模式映像
```

保证。

选择题高频：

```text
外模式/模式映像 → 逻辑独立性
模式/内模式映像 → 物理独立性
```

---

## 1.4 三级模式结构

数据库系统的三级模式结构：

```text
外模式
模式 / 概念模式 / 逻辑模式
内模式
```

### 1.4.1 外模式

外模式也叫用户模式、子模式，是用户能够看到和使用的局部数据的逻辑结构。

关键词：

```text
用户视图
局部数据
不同用户可以有不同外模式
```

考试答法：

```text
外模式是数据库用户能够看见和使用的局部数据的逻辑结构和特征的描述，是数据库用户的数据视图。
```

---

### 1.4.2 模式 / 概念模式

模式是数据库中全体数据的逻辑结构和特征的描述。

关键词：

```text
全局逻辑结构
全体数据
一个数据库只有一个模式
```

考试答法：

```text
模式是数据库中全体数据的逻辑结构和特征的描述，是所有用户公共的数据视图。
```

选择题：

```text
描述整个数据库全局逻辑视图的是：概念模式 / 模式
```

---

### 1.4.3 内模式

内模式是数据在数据库内部的物理存储结构和存取方式的描述。

关键词：

```text
物理存储
存取方式
文件组织
索引
```

考试答法：

```text
内模式描述数据在数据库内部的物理存储结构和存取方法。
```

---

## 1.5 两级映像

三级模式之间有两级映像：

```text
外模式 / 模式映像
模式 / 内模式映像
```

### 外模式 / 模式映像

作用：

```text
保证逻辑独立性
```

意思是：

```text
模式改变时，可以通过修改外模式/模式映像，使外模式和应用程序尽量不变。
```

### 模式 / 内模式映像

作用：

```text
保证物理独立性
```

意思是：

```text
内模式改变时，可以通过修改模式/内模式映像，使模式和应用程序尽量不变。
```

---

## 1.6 数据模型

数据模型是现实世界数据特征的抽象。

数据模型三要素：

```text
数据结构
数据操作
完整性约束
```

### 1.6.1 数据结构

描述系统的静态特性。

例如关系模型中的表、属性、元组。

### 1.6.2 数据操作

描述系统的动态特性。

例如查询、插入、删除、修改。

### 1.6.3 完整性约束

保证数据正确、有效、相容。

例如主码不能为空、外码必须引用已有记录、成绩必须在 0 到 100 之间。

---

## 1.7 概念模型与 E-R 图

### 1.7.1 基本概念

```text
实体：客观存在并可相互区别的事物
属性：实体具有的特征
码：能唯一标识实体的属性或属性组
实体型：同类实体的抽象
实体集：同型实体的集合
联系：实体之间的关系
```

### 1.7.2 E-R 图符号

```text
矩形：实体
椭圆：属性
菱形：联系
线段：连接实体、属性、联系
```

选择题：

```text
E-R 图中菱形表示联系。
```

### 1.7.3 联系类型

```text
一对一 1:1
一对多 1:n
多对多 m:n
递归联系
```

#### 递归联系

递归联系是同一个实体集内部实体之间的联系。

例子：

```text
员工 管理 员工
课程 先修 课程
零件 组成 零件
```

---

## 1.8 关系模型基本术语

这部分选择题非常爱考。

```text
关系 = 一张表
元组 = 表中的一行
属性 = 表中的一列
分量 = 某一行某一列的一个值
域 = 属性的取值范围
码 = 能唯一标识元组的属性或属性组
候选码 = 最小的超码
主码 = 从候选码中选定的一个码
主属性 = 包含在任一候选码中的属性
非主属性 = 不包含在任何候选码中的属性
```

例子：

| Sno | Sname | Sex | Age |
|---|---|---|---|
| 001 | 张三 | 男 | 20 |
| 002 | 李四 | 女 | 21 |

解释：

```text
整张 Student 表：关系
001 张三 男 20：一个元组
Sno、Sname、Sex、Age：属性
“张三”：一个分量
Sex 的取值范围 {男,女}：域
```

---

## 1.9 关系的性质

常考：

```text
1. 列是同质的，即每一列来自同一个域
2. 不同列可以来自同一个域，但属性名必须不同
3. 列的顺序无关
4. 行的顺序无关
5. 任意两个元组不能完全相同
6. 每个分量必须是不可再分的数据项
```

最后一点对应：

```text
1NF：属性值不可再分
```

---

## 1.10 关系代数基础

常见关系代数符号：

```text
σ：选择，选行，对应 SQL 的 WHERE
Π：投影，选列，对应 SQL 的 SELECT 列名
×：笛卡尔积
⋈：连接
∪：并
-：差
∩：交
÷：除
```

### 选择 σ

选满足条件的行。

```text
σ Sex='男'(Student)
```

SQL：

```sql
SELECT *
FROM Student
WHERE Sex='男';
```

### 投影 Π

选指定列。

```text
Π Sname,Age(Student)
```

SQL：

```sql
SELECT Sname, Age
FROM Student;
```

### 除法 ÷

看到：

```text
选修了全部课程的学生
销售所有商品的市场
参加了所有培训的员工
```

关系代数一般用除法，SQL 常用双重 `NOT EXISTS`。

# 2. 数据库安全性

## 2.1 安全性是什么

数据库安全性是指保护数据库，防止非法用户使用数据库，以及合法用户越权访问数据库。

关键词：

```text
防止非法访问
防止越权使用
保护数据不被泄露、篡改、破坏
```

考试答法：

```text
数据库安全性是指保护数据库以防止不合法使用所造成的数据泄露、更改或破坏，保证只有被授权的用户才能在授权范围内访问数据库。
```

---

## 2.2 安全性控制的主要方法

常见方法：

```text
1. 用户标识与鉴别
2. 存取控制
3. 自主存取控制 DAC
4. 强制存取控制 MAC
5. 视图机制
6. 审计
7. 数据加密
```

选择题：

```text
出入机房登记是物理管理措施，不属于数据库系统内部的安全性控制主要技术。
```

---

## 2.3 用户标识与鉴别

用户访问数据库前，系统要确认用户身份。

常见方式：

```text
用户名 + 密码
身份认证
口令验证
生物特征
```

考试答法：

```text
用户标识与鉴别是系统识别用户身份、判断其是否为合法用户的安全措施，是数据库安全保护的第一道防线。
```

---

## 2.4 存取控制

存取控制是数据库安全性控制的核心。

它要解决：

```text
谁可以访问哪些数据
可以进行什么操作
```

常见权限：

```text
SELECT：查询
INSERT：插入
UPDATE：修改
DELETE：删除
REFERENCES：引用
ALL PRIVILEGES：全部权限
```

授权语句：

```sql
GRANT SELECT, UPDATE
ON Student
TO user1;
```

收回权限：

```sql
REVOKE SELECT, UPDATE
ON Student
FROM user1;
```

### GRANT

`GRANT` 是授权，属于 DCL。

```sql
GRANT SELECT
ON Student
TO zhangsan;
```

表示把 Student 表的查询权限授予 zhangsan。

### REVOKE

`REVOKE` 是收回权限，属于 DCL。

```sql
REVOKE SELECT
ON Student
FROM zhangsan;
```

选择题：

```text
GRANT / REVOKE 属于数据控制语言 DCL。
```

---

## 2.5 自主存取控制 DAC

自主存取控制是指用户可以把自己拥有的权限授予其他用户。

关键词：

```text
用户自主授权
GRANT
REVOKE
灵活
```

常见风险：

```text
权限传播可能导致安全隐患
```

---

## 2.6 强制存取控制 MAC

强制存取控制是系统根据主体和客体的安全级别进行访问控制。

主体：

```text
用户
进程
事务
```

客体：

```text
表
视图
元组
属性
```

常见安全级别：

```text
绝密
机密
秘密
公开
```

特点：

```text
不是用户自主决定，而是系统强制控制
安全性更高
```

---

## 2.7 视图机制与安全性

视图可以隐藏不希望用户看到的数据。

例如学生表：

```text
Student(Sno, Sname, Sex, Age, Phone, Address)
```

只允许某用户看姓名和性别：

```sql
CREATE VIEW V_Student
AS
SELECT Sname, Sex
FROM Student;
```

再授权：

```sql
GRANT SELECT
ON V_Student
TO user1;
```

作用：

```text
用户只能看到视图中的列，看不到基本表中的敏感信息。
```

考试答法：

```text
视图机制通过为不同用户定义不同视图，只暴露其需要访问的数据，从而隐藏敏感数据，增强数据库安全性。
```

---

## 2.8 审计

审计是把用户对数据库的操作记录下来，以便事后检查和追踪。

审计内容：

```text
谁
在什么时候
对什么数据
执行了什么操作
是否成功
```

作用：

```text
发现非法操作
追踪责任
提高安全性
```

考试答法：

```text
审计是 DBMS 将用户对数据库的操作自动记录到审计日志中，以便事后检查、分析和追踪非法访问行为的一种安全控制方法。
```

---

## 2.9 数据加密

数据加密是将明文数据转换为密文，防止数据泄露。

可用于：

```text
存储加密
传输加密
口令加密
```

---

## 2.10 安全性与完整性的区别

这个很容易混。

| 对比 | 安全性 | 完整性 |
|---|---|---|
| 关注点 | 谁能访问数据 | 数据是否正确有效 |
| 目的 | 防止非法访问、越权访问 | 防止错误数据进入数据库 |
| 例子 | 用户没有权限不能查工资 | 成绩必须在 0 到 100 之间 |
| 技术 | 授权、视图、审计、加密 | 主码、外码、CHECK、触发器 |

一句话：

```text
安全性管“谁能用”，完整性管“数据对不对”。
```

# 3. 数据库完整性

## 3.1 完整性是什么

数据库完整性是指数据库中数据的正确性、有效性和相容性。

关键词：

```text
正确性
有效性
相容性
```

考试答法：

```text
数据库完整性是指数据库中数据的正确性、有效性和相容性。完整性控制用于防止不符合语义的数据进入数据库。
```

---

## 3.2 关系模型的三类完整性约束

关系模型有三类完整性约束：

```text
实体完整性
参照完整性
用户自定义完整性
```

选择题：

```text
对象完整性不是关系模型的三类完整性。
```

---

## 3.3 实体完整性

实体完整性规则：

```text
主码属性不能为空。
```

原因：

```text
主码用于唯一标识元组。如果主码为空，就无法区分实体。
```

例子：

```text
Student(Sno, Sname, Sex)
Sno 是主码
Sno 不能为 NULL
```

考试答法：

```text
实体完整性要求关系中的主属性不能为空，尤其主码不能取空值，因为主码用于唯一标识关系中的元组。
```

注意：

```text
联合主码中的每个属性都不能为空。
```

例如：

```text
SC(Sno, Cno, Grade)
主码是 (Sno, Cno)
则 Sno 和 Cno 都不能空。
```

---

## 3.4 参照完整性

参照完整性规则：

```text
外码的取值要么为空，要么等于被参照关系中某个主码值。
```

例子：

```text
Student(Sno, Sname)
SC(Sno, Cno, Grade)
```

`SC.Sno` 是外码，引用 `Student.Sno`。

那么 `SC.Sno` 不能随便填一个学生表中不存在的学号。

考试答法：

```text
参照完整性用于描述关系之间的引用约束。外码的取值必须为空或等于被参照关系中某个元组的主码值。
```

---

## 3.5 用户自定义完整性

用户自定义完整性是针对具体应用语义定义的约束。

例子：

```text
成绩必须在 0 到 100 之间
性别只能为 男 或 女
年龄必须大于 0
学号必须是 12 位数字
工资不能为负
```

考试答法：

```text
用户自定义完整性是针对某一具体应用领域的数据语义约束，由用户根据实际业务需要定义。
```

选择题：

```text
“学号必须由 12 个数字组成”属于用户自定义完整性。
```

---

## 3.6 SQL 中的完整性约束

### 主码约束 PRIMARY KEY

```sql
CREATE TABLE Student (
    Sno CHAR(12) PRIMARY KEY,
    Sname VARCHAR(20)
);
```

联合主码：

```sql
CREATE TABLE SC (
    Sno CHAR(12),
    Cno CHAR(6),
    Grade INT,
    PRIMARY KEY (Sno, Cno)
);
```

### 唯一约束 UNIQUE

```sql
Email VARCHAR(50) UNIQUE
```

表示 Email 不能重复。

注意：

```text
UNIQUE 和 PRIMARY KEY 都能保证唯一；
但 PRIMARY KEY 不能空，UNIQUE 在不同 DBMS 中对 NULL 的处理可能不同。
```

### 非空约束 NOT NULL

```sql
Sname VARCHAR(20) NOT NULL
```

### 外码约束 FOREIGN KEY

```sql
FOREIGN KEY (Sno) REFERENCES Student(Sno)
```

### 检查约束 CHECK

```sql
CHECK (Grade >= 0 AND Grade <= 100)
```

---

## 3.7 完整性约束的违约处理

当插入、删除、修改违反完整性约束时，DBMS 会拒绝操作或执行相应处理。

例如：

```text
插入一个不存在学生的选课记录 → 违反参照完整性
删除一个仍被选课表引用的学生 → 可能违反参照完整性
把成绩改成 150 → 违反用户自定义完整性
```

外码相关处理方式可能有：

```text
拒绝删除/修改
级联删除/修改
设置为空值
设置为默认值
```

---

## 3.8 安全性控制 vs 完整性控制

概念题可以这样答：

```text
安全性控制是防止非法用户访问数据库或合法用户越权访问数据库；
完整性控制是防止不符合语义、不正确或无效的数据进入数据库。
二者都是 DBMS 统一管理和控制数据库的重要功能。
```

---

# 4. 数据库恢复技术

## 4.1 为什么需要恢复

数据库运行过程中可能发生故障，导致数据库处于不一致状态。

数据库恢复的目的：

```text
把数据库从错误状态恢复到某个已知的正确状态
保证事务的原子性和持久性
```

考试答法：

```text
数据库恢复是指当数据库发生故障后，利用备份和日志等手段，将数据库恢复到某个一致性状态的过程。
```

---

## 4.2 事务

事务是用户定义的一个数据库操作序列，这些操作要么全做，要么全不做，是数据库恢复和并发控制的基本单位。

考试答法：

```text
事务是数据库系统中一个不可分割的工作单位，是一组 SQL 操作序列。事务中的操作要么全部成功，要么全部失败。
```

例子：

```text
银行转账：
A 账户扣 100
B 账户加 100
```

这两个操作必须作为一个事务。

---

## 4.3 事务 ACID 特性

事务有四个特性：

```text
Atomicity 原子性
Consistency 一致性
Isolation 隔离性
Durability 持久性
```

### 4.3.1 原子性 Atomicity

事务是不可分割的单位，要么全部执行，要么全部不执行。

失败时要回滚。

关键词：

```text
全做或全不做
UNDO 保证
```

---

### 4.3.2 一致性 Consistency

事务执行前后，数据库都必须从一个一致性状态变到另一个一致性状态。

例如：

```text
转账前 A+B 总金额为 1000
转账后 A+B 总金额仍为 1000
```

---

### 4.3.3 隔离性 Isolation

多个事务并发执行时，一个事务的执行不应被其他事务干扰。

效果上好像事务一个一个串行执行。

并发控制保证隔离性。

---

### 4.3.4 持久性 Durability

事务一旦提交，它对数据库的改变就是永久的，即使之后系统故障也不能丢失。

恢复机制和 REDO 保证持久性。

---

## 4.4 事务状态

常见事务状态：

```text
活动状态
部分提交状态
提交状态
失败状态
中止状态
```

### 活动状态

事务正在执行。

### 部分提交状态

事务最后一条语句执行完，但提交操作还没完全完成。

### 提交状态

事务成功提交。

考试注意：

```text
最后一条语句执行完：部分提交
commit 成功：提交状态 / 已提交状态
```

### 失败状态

事务不能继续正常执行。

### 中止状态

事务回滚完毕，撤销了所有影响。

---

## 4.5 COMMIT 与 ROLLBACK

### COMMIT

提交事务，使事务修改永久生效。

```sql
COMMIT;
```

### ROLLBACK

回滚事务，撤销事务已做的修改。

```sql
ROLLBACK;
```

选择题：

```text
事务回滚语句：ROLLBACK
事务提交语句：COMMIT
```

---

## 4.6 故障类型

常见数据库故障：

```text
事务故障
系统故障
介质故障
计算机病毒
```

### 4.6.1 事务故障

某个事务自身失败。

原因：

```text
程序错误
输入错误
违反完整性约束
死锁被撤销
用户主动回滚
```

恢复：

```text
UNDO 该事务
```

---

### 4.6.2 系统故障

系统停止运行，如断电、操作系统崩溃、DBMS 崩溃。

特点：

```text
内存数据丢失
磁盘数据一般没坏
```

恢复：

```text
对已提交但可能未写入磁盘的事务 REDO
对未提交事务 UNDO
```

---

### 4.6.3 介质故障

外存损坏，如磁盘坏了。

特点：

```text
数据库文件可能损坏
危害最大
```

恢复：

```text
装入数据库备份
利用日志重做已提交事务
```

---

### 4.6.4 计算机病毒

恶意程序破坏数据库。

应对：

```text
安全控制
备份
恢复
审计
```

---

## 4.7 恢复的基本技术

恢复技术主要依靠：

```text
数据转储 / 备份
日志文件
检查点
REDO / UNDO
```

---

## 4.8 数据转储

数据转储就是定期把数据库复制到备用存储介质上，形成备份。

分类：

```text
静态转储
动态转储
海量转储
增量转储
```

### 静态转储

系统中无事务运行时进行转储。

优点：

```text
简单，备份数据一致
```

缺点：

```text
需要停止事务，影响系统运行
```

### 动态转储

转储期间允许事务运行。

优点：

```text
不影响数据库运行
```

缺点：

```text
备份期间数据库可能变化，需要配合日志恢复
```

### 海量转储

每次转储全部数据库。

### 增量转储

只转储上次转储后发生变化的数据。

---

## 4.9 日志文件

日志文件记录事务对数据库的更新操作，是恢复的重要依据。

日志内容常见格式：

```text
<Ti start>
<Ti, X, old_value, new_value>
<Ti commit>
<Ti abort>
```

含义：

```text
<Ti start>：事务 Ti 开始
<Ti, X, old, new>：Ti 把 X 从 old 改为 new
<Ti commit>：Ti 提交
<Ti abort>：Ti 中止
```

日志作用：

```text
进行事务故障恢复
进行系统故障恢复
协助介质故障恢复
```

---

## 4.10 日志恢复规则：REDO 和 UNDO

### REDO

REDO 是重做。

用于保证已提交事务的结果一定反映到数据库中。

```text
REDO：写新值 new_value
```

例子：

```text
<T1, A, 100, 200>
REDO T1：A = 200
```

### UNDO

UNDO 是撤销。

用于撤销未提交事务对数据库的影响。

```text
UNDO：写旧值 old_value
```

例子：

```text
<T2, B, 50, 80>
UNDO T2：B = 50
```

核心口诀：

```text
有 commit → REDO → 写新值
无 commit → UNDO → 写旧值
```

---

## 4.11 日志题判断流程

给一段日志，让你判断 REDO 集和 UNDO 集。

### 无检查点情况

步骤：

```text
1. 从头到尾看日志
2. 有 start 说明事务开始
3. 有 commit 说明事务已提交
4. 崩溃时已经 commit 的事务进 REDO
5. 崩溃时没有 commit 的事务进 UNDO
```

例子：

```text
<T1 start>
<T1, A, 100, 200>
<T2 start>
<T2, B, 50, 60>
<T1 commit>
<T3 start>
<T3, C, 10, 20>
crash
```

判断：

```text
T1 有 commit → REDO
T2 无 commit → UNDO
T3 无 commit → UNDO
```

答案：

```text
REDO = {T1}
UNDO = {T2, T3}
```

恢复操作：

```text
REDO T1：A = 200
UNDO T2：B = 50
UNDO T3：C = 10
```

---

## 4.12 有检查点 checkpoint 的日志题

检查点记录某一时刻正在执行的事务。

格式可能是：

```text
<checkpoint {T2, T3}>
```

表示检查点时 T2、T3 正在执行。

处理步骤：

```text
1. 找最后一个 checkpoint
2. checkpoint 中活跃事务先放入 UNDO
3. 从 checkpoint 后继续扫描
4. 遇到 start，把事务加入 UNDO
5. 遇到 commit，把事务从 UNDO 移到 REDO
6. 最后 REDO 已提交事务，UNDO 未提交事务
```

例子：

```text
<T1 start>
<T1, A, 100, 200>
<T1 commit>
<T2 start>
<T2, B, 50, 60>
<checkpoint {T2}>
<T3 start>
<T3, C, 10, 20>
<T2 commit>
crash
```

从 checkpoint 开始：

```text
checkpoint 时 T2 活跃 → UNDO={T2}
T3 start → UNDO={T2,T3}
T2 commit → T2 从 UNDO 移到 REDO
```

最后：

```text
REDO={T2}
UNDO={T3}
```

T1 在检查点前已经提交，通常不需要处理。

---

## 4.13 恢复策略总结

| 故障类型 | 恢复方法 |
|---|---|
| 事务故障 | UNDO 该事务 |
| 系统故障 | UNDO 未提交事务，REDO 已提交事务 |
| 介质故障 | 恢复备份，再 REDO 日志 |
| 病毒破坏 | 使用备份和日志恢复，加强安全控制 |

---

## 4.14 检查点的作用

检查点可以减少恢复时需要扫描的日志范围，提高恢复效率。

考试答法：

```text
检查点记录某一时刻数据库的一致状态和正在运行的事务。发生故障后，系统可从最近的检查点开始恢复，减少需要扫描的日志记录，提高恢复效率。
```

---

# 5. 并发控制

## 5.1 为什么需要并发控制

多用户数据库系统中，多个事务可能同时访问同一数据。

并发执行优点：

```text
提高系统资源利用率
提高事务吞吐量
减少等待时间
```

但如果不控制，会产生数据不一致问题。

并发控制的目的：

```text
保证事务的隔离性
保证数据库一致性
使并发执行效果等价于某种串行执行
```

考试答法：

```text
并发控制是 DBMS 对并发事务的执行进行控制，以避免数据不一致，保证事务隔离性和数据库一致性的机制。
```

---

## 5.2 并发操作带来的数据不一致问题

常见问题：

```text
丢失修改 / 丢失更新
读脏数据
不可重复读 / 读值不可重现
幻读
```

选择题注意：

```text
修改异常属于关系模式设计问题，不是并发操作造成的数据不一致。
```

---

## 5.3 丢失修改 / 丢失更新

两个事务同时修改同一数据，后一个事务的修改覆盖前一个事务的修改。

例子：

```text
A = 100

T1 读 A=100
T2 读 A=100
T1 A=A-10，写回 90
T2 A=A-20，写回 80
```

正确结果应该是 70，但最后是 80，T1 的修改丢失。

关键词：

```text
两个事务修改同一数据
一个修改覆盖另一个修改
```

---

## 5.4 读脏数据

一个事务读到了另一个事务尚未提交的数据。

例子：

```text
T1 把 A 从 100 改成 200，但未提交
T2 读取 A=200
T1 回滚，A 又变回 100
```

T2 读到的 200 就是脏数据。

关键词：

```text
读到未提交数据
对方后来回滚
```

---

## 5.5 不可重复读 / 读值不可重现

一个事务两次读取同一数据，中间被其他事务修改并提交，导致两次读到的值不同。

例子：

```text
T1 第一次读 A=100
T2 修改 A=200 并提交
T1 第二次读 A=200
```

关键词：

```text
同一事务两次读取同一数据
结果不同
```

---

## 5.6 幻读

一个事务按条件查询某些记录，另一个事务插入或删除了满足条件的记录，导致前后两次查询的记录数不同。

例子：

```text
T1 查询所有工资 > 5000 的员工，有 10 人
T2 插入一个工资 6000 的员工并提交
T1 再查，有 11 人
```

关键词：

```text
同一范围查询
前后多出或少了记录
```

---

## 5.7 封锁技术

封锁是并发控制的常用方法。

定义：

```text
事务在访问某数据对象前，先向系统申请对该对象加锁，获得锁后才能访问。
```

常见锁：

```text
排他锁 X 锁 / 写锁
共享锁 S 锁 / 读锁
```

---

## 5.8 X 锁

X 锁是排他锁，也叫写锁。

如果事务 T 对数据 A 加 X 锁，则其他事务不能再对 A 加任何锁，直到 T 释放 X 锁。

作用：

```text
用于修改数据
防止其他事务读写该数据
```

兼容性：

```text
X 与 X 不兼容
X 与 S 不兼容
```

---

## 5.9 S 锁

S 锁是共享锁，也叫读锁。

如果事务 T 对数据 A 加 S 锁，则其他事务还可以对 A 加 S 锁，但不能加 X 锁。

作用：

```text
用于读取数据
允许多个事务同时读
禁止其他事务写
```

兼容性：

```text
S 与 S 兼容
S 与 X 不兼容
```

选择题：

```text
一个事务对 A 加了 S 锁，其他事务还能对 A 加 S 锁。
```

---

## 5.10 S/X 锁兼容表

| 已有锁 \ 请求锁 | S 锁 | X 锁 |
|---|---|---|
| S 锁 | 兼容 | 不兼容 |
| X 锁 | 不兼容 | 不兼容 |

记忆：

```text
读读可以，读写不行，写写不行。
```

---

## 5.11 三级封锁协议

### 一级封锁协议

规则：

```text
事务在修改数据前必须加 X 锁，直到事务结束才释放。
```

解决：

```text
丢失修改 / 丢失更新
```

不能解决：

```text
读脏数据
不可重复读
```

---

### 二级封锁协议

规则：

```text
在一级封锁协议基础上，事务读取数据前必须加 S 锁，读完即可释放 S 锁。
```

解决：

```text
丢失修改
读脏数据
```

原因：

```text
读数据前加 S 锁，如果其他事务正加 X 锁修改未提交，则读事务不能读到未提交数据。
```

不能解决：

```text
不可重复读
```

因为 S 锁读完就释放，别人之后还可以修改。

---

### 三级封锁协议

规则：

```text
在一级封锁协议基础上，事务读取数据前必须加 S 锁，直到事务结束才释放。
```

解决：

```text
丢失修改
读脏数据
不可重复读
```

选择题注意：

```text
一级 → 防丢失修改
二级 → 防丢失修改 + 读脏数据
三级 → 防丢失修改 + 读脏数据 + 不可重复读
```

---

## 5.12 活锁与死锁

### 活锁

事务一直等待，但系统一直没有调度它执行，导致它长期得不到资源。

解决方法：

```text
先来先服务
公平调度
```

### 死锁

多个事务互相等待对方释放资源，导致都无法继续执行。

例子：

```text
T1 锁住 A，等待 B
T2 锁住 B，等待 A
```

---

## 5.13 死锁检测：等待图法

等待图：

```text
结点：事务
边：等待关系
Ti -> Tj 表示 Ti 等待 Tj 释放资源
```

判断：

```text
等待图中出现环 / 回路，则发生死锁。
```

考试答法：

```text
可采用等待图法检测死锁。用结点表示事务，若事务 Ti 等待事务 Tj 释放资源，则画 Ti 指向 Tj 的边。若等待图中存在回路，则说明系统发生死锁。
```

---

## 5.14 死锁解除

方法：

```text
撤销或回滚一个或多个事务
选择牺牲事务
释放其占用的资源
超时法
抢占资源
```

考试答法：

```text
发生死锁后，可以选择一个或多个代价较小的事务作为牺牲事务，将其撤销或回滚，释放其占用的资源，从而解除死锁。
```

---

## 5.15 死锁预防

常见方法：

```text
一次封锁法
顺序封锁法
```

### 一次封锁法

事务开始前一次性申请所有需要的锁。

缺点：

```text
降低并发度
难以预知事务需要的全部数据
```

### 顺序封锁法

规定数据对象的封锁顺序，所有事务按同一顺序加锁。

作用：

```text
防止循环等待，从而预防死锁
```

---

## 5.16 两段锁协议 2PL

两段锁协议是保证并发调度可串行化的重要协议。

规则：

```text
事务分为两个阶段：
1. 扩展阶段：只能申请锁，不能释放锁
2. 收缩阶段：只能释放锁，不能再申请锁
```

考试答法：

```text
两段锁协议规定事务的封锁过程分为扩展阶段和收缩阶段。扩展阶段只能获得锁不能释放锁；收缩阶段只能释放锁不能再获得锁。遵守两段锁协议的并发调度是可串行化的。
```

选择题：

```text
DBMS 采用两段锁协议保证并发调度的可串行化。
```

注意：

```text
两段锁协议可以保证可串行化，但不能保证不发生死锁。
```

---

## 5.17 可串行化调度

多个事务并发执行，如果其执行结果与某个串行执行结果相同，则称该调度是可串行化的。

考试答法：

```text
可串行化调度是指多个事务并发执行的结果与这些事务按某种串行顺序执行的结果相同的调度。
```

并发控制的正确性标准：

```text
可串行化
```

## 5.18 并发控制简答模板

### 问：为什么需要并发控制？

```text
多个事务并发执行可以提高系统效率，但若不加控制，可能产生丢失修改、读脏数据、不可重复读等数据不一致问题。因此 DBMS 需要并发控制，以保证事务隔离性和数据库一致性，使并发执行结果等价于某种串行执行。
```

### 问：什么是封锁？

```text
封锁是并发控制的常用技术。事务在访问某数据对象前，必须先向系统申请对该对象加锁，获得相应锁后才能访问。常见锁有共享锁 S 锁和排他锁 X 锁。
```

### 问：S 锁和 X 锁区别？

```text
S 锁又称共享锁或读锁。若事务对数据对象加 S 锁，其他事务还可以再加 S 锁，但不能加 X 锁。
X 锁又称排他锁或写锁。若事务对数据对象加 X 锁，其他事务不能再对该对象加任何锁。
```

### 问：两段锁协议是什么？

```text
两段锁协议规定事务的加锁和解锁分为两个阶段：扩展阶段只能申请锁不能释放锁，收缩阶段只能释放锁不能再申请锁。遵守两段锁协议的调度是可串行化的。
```

### 问：死锁如何检测和解除？

```text
可采用等待图法检测死锁。等待图中结点表示事务，边表示等待关系，若图中存在回路，则说明发生死锁。解除死锁可选择一个或多个事务作为牺牲事务，将其撤销或回滚，释放资源。
```

## 6.2 安全性与完整性

```text
安全性：防止非法访问、越权访问。
完整性：保证数据正确、有效、相容。
实体完整性：主码不能为空。

```

## 6.3 恢复技术

```text
事务 ACID：原子性、一致性、隔离性、持久性。
没有“唯一性”“可靠性”这种事务性质。
COMMIT：提交。
ROLLBACK：回滚。
REDO：已提交事务，写新值。
UNDO：未提交事务，写旧值。
事务故障：UNDO。
系统故障：UNDO 未提交，REDO 已提交。
介质故障：备份 + 日志。
检查点：减少恢复扫描日志范围。
```

## 6.4 并发控制

```text
并发问题：丢失修改、读脏数据、不可重复读、幻读。

S 锁：读锁，共享锁，S-S 兼容。
X 锁：写锁，排他锁，与 S/X 都不兼容。
一级封锁协议：防丢失修改。
二级封锁协议：防丢失修改、读脏数据。
三级封锁协议：防丢失修改、读脏数据、不可重复读。
两段锁协议：保证可串行化，但可能死锁。
等待图有环：死锁。

```

# 7. 概念大题速背模板

## 7.1 数据库系统特点

```text
数据库系统具有数据结构化、数据共享性高、冗余度低、数据独立性高、由 DBMS 统一管理和控制等特点。DBMS 提供安全性控制、完整性控制、并发控制和恢复功能，从而保证数据安全、正确和可靠。
```

---

## 7.2 三级模式结构及优点

```text
数据库系统的三级模式结构包括外模式、模式和内模式。外模式是用户能够看到和使用的局部数据视图；模式是数据库全体数据的逻辑结构和特征的描述；内模式是数据的物理存储结构和存取方式的描述。三级模式结构通过外模式/模式映像和模式/内模式映像分别保证逻辑独立性和物理独立性，提高了数据独立性，也有利于数据安全管理。
```

---

## 7.3 数据库安全性

```text
数据库安全性是指保护数据库，防止非法用户访问数据库以及合法用户越权访问数据库，避免数据被泄露、篡改或破坏。常用安全性控制方法包括用户标识与鉴别、存取控制、授权与收回权限、视图机制、审计和数据加密等。
```

---

## 7.4 数据库完整性

```text
数据库完整性是指数据库中数据的正确性、有效性和相容性。关系模型的完整性约束包括实体完整性、参照完整性和用户自定义完整性。实体完整性要求主码不能为空；参照完整性要求外码取值为空或引用被参照关系中的主码值；用户自定义完整性是根据具体应用语义定义的约束。
```

---

## 7.5 事务 ACID

```text
事务是数据库系统中一个不可分割的工作单位，是一组 SQL 操作序列。事务具有 ACID 特性：原子性表示事务中的操作要么全做要么全不做；一致性表示事务执行前后数据库都保持一致状态；隔离性表示并发事务之间互不干扰；持久性表示事务一旦提交，其对数据库的修改就是永久的。
```

---

## 7.6 数据库恢复

```text
数据库恢复是指当数据库发生故障后，利用备份和日志等手段，将数据库恢复到某个一致性状态的过程。恢复技术主要包括数据转储、日志文件、检查点以及 REDO 和 UNDO 操作。对已提交事务需要 REDO 以保证持久性，对未提交事务需要 UNDO 以保证原子性。
```

---

## 7.7 并发控制

```text
并发控制是 DBMS 对多个并发事务的执行进行控制，以避免数据不一致，保证事务隔离性和数据库一致性的机制。并发操作可能带来丢失修改、读脏数据、不可重复读等问题。常用并发控制方法是封锁技术，包括共享锁 S 锁和排他锁 X 锁。两段锁协议可以保证并发调度的可串行化。
```

---

## 7.8 死锁检测与解除

```text
可以采用等待图法检测死锁。等待图中结点表示事务，若事务 Ti 等待事务 Tj 释放资源，则画 Ti 指向 Tj 的边。若等待图中存在回路，则说明发生死锁。发生死锁后，可以选择一个或多个代价较小的事务作为牺牲事务，将其撤销或回滚，释放其占用的资源，从而解除死锁。
```

# 8. 日志 REDO / UNDO 题万能模板

## 8.1 无检查点

```text
1. 扫描日志，找出所有事务。
2. 有 commit 的事务加入 REDO 集。
3. 没有 commit 的事务加入 UNDO 集。
4. REDO 写新值，UNDO 写旧值。
```

模板答案：

```text
T1 已提交，因此需要 REDO；
T2 在系统故障时尚未提交，因此需要 UNDO。
REDO 操作按照日志记录写入新值，UNDO 操作按照日志记录恢复旧值。
```

---

## 8.2 有检查点

```text
1. 找最后一个 checkpoint。
2. checkpoint 中活跃事务先加入 UNDO。
3. 从 checkpoint 后向后扫描日志。
4. 遇到 start 加入 UNDO。
5. 遇到 commit 从 UNDO 移到 REDO。
6. 最后 REDO 已提交事务，UNDO 未提交事务。
```

模板答案：

```text
从最后一个检查点开始分析。检查点时正在执行的事务加入 UNDO 集；检查点之后开始但未提交的事务也加入 UNDO 集；检查点之后提交的事务加入 REDO 集。恢复时，对 REDO 集事务重做其更新，对 UNDO 集事务撤销其更新。
```
