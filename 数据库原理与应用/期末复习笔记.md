# 第一章 绪论

## 1.1 数据库相关基本概念

### 1. 数据 Data

数据是描述事物的符号记录。  
例如：学生姓名、学号、成绩、课程号等。

### 2. 数据库 DB，Database

数据库是**长期存储在计算机内、有组织、可共享的大量数据集合**。

关键词：

- 长期存储
- 有组织
- 可共享
- 数据集合

### 3. 数据库管理系统 DBMS

数据库管理系统是位于用户和操作系统之间的一层数据管理软件。

主要功能：

1. 数据定义功能：定义数据库结构，例如建表。
2. 数据组织、存储和管理功能。
3. 数据操纵功能：查询、插入、删除、修改。
4. 数据库事务管理和运行管理。
5. 数据库建立和维护功能。
6. 数据通信接口。

常见 DBMS：MySQL、SQL Server、Oracle、PostgreSQL、SQLite。

### 4. 数据库系统 DBS

数据库系统一般由以下部分组成：

```text
数据库系统 DBS = 数据库 DB + 数据库管理系统 DBMS + 应用程序 + 数据库管理员 DBA + 用户
```

注意区分：

| 名称 | 含义 |
|---|---|
| DB | 数据库本身，是数据集合 |
| DBMS | 管理数据库的软件 |
| DBS | 数据库系统，是完整系统 |
| DBA | 数据库管理员 |

---

## 1.2 数据管理技术的发展阶段

### 1. 人工管理阶段

特点：

- 数据不保存。
- 应用程序直接管理数据。
- 数据不共享。
- 数据冗余大。
- 数据和程序高度依赖。

### 2. 文件系统阶段

特点：

- 数据可以长期保存。
- 由文件系统管理数据。
- 数据共享性差。
- 数据冗余和不一致问题仍然明显。
- 数据独立性差。

### 3. 数据库系统阶段

特点：

- 数据结构化。
- 数据共享性高、冗余度低。
- 数据独立性高。
- 数据由 DBMS 统一管理和控制。
- 支持并发控制、恢复、安全性、完整性。

---

## 1.3 数据库系统的特点

数据库系统的主要特点：

1. **数据结构化**：数据库中的数据是有组织、有结构的。
2. **数据共享性高、冗余度低**：多个应用可以共享同一份数据。
3. **数据独立性高**：数据结构变化时，应用程序尽量少改。
4. **数据由 DBMS 统一管理和控制**：包括安全性、完整性、并发控制、恢复。

---

## 1.4 数据独立性

数据独立性是数据库系统的重要特点。

### 1. 物理独立性

物理独立性指：

> 数据的物理存储方式改变时，应用程序和逻辑结构尽量不变。

例如：数据文件换了存储位置、索引方式改变，用户程序不需要改。

### 2. 逻辑独立性

逻辑独立性指：

> 数据库的逻辑结构改变时，应用程序尽量不变。

例如：表中增加一个字段，原有程序尽量不受影响。

一般来说：

```text
逻辑独立性比物理独立性更难实现。
```

---

## 1.5 数据库系统的三级模式结构

三级模式结构：

```text
外模式  —— 用户看到的数据视图
模式    —— 数据库的全局逻辑结构
内模式  —— 数据的物理存储结构
```

### 1. 外模式 External Schema

外模式也叫用户模式，是用户能看到和使用的局部数据逻辑结构。

特点：

- 一个数据库可以有多个外模式。
- 外模式面向具体用户或应用。
- 视图通常属于外模式层面的内容。

### 2. 模式 Schema

模式也叫概念模式，是数据库中全体数据的逻辑结构和特征描述。

特点：

- 一个数据库只有一个模式。
- 它描述全局逻辑结构。

### 3. 内模式 Internal Schema

内模式也叫存储模式，描述数据在数据库中的物理存储方式。

特点：

- 一个数据库只有一个内模式。
- 关注数据如何存储，例如文件组织、索引等。

---

## 1.6 两级映像

两级映像用于保证数据独立性。

```text
外模式 / 模式映像：保证逻辑独立性
模式 / 内模式映像：保证物理独立性
```

### 1. 外模式 / 模式映像

当模式改变时，只需要修改外模式/模式映像，外模式可以不变，从而应用程序尽量不变。

作用：保证**逻辑独立性**。

### 2. 模式 / 内模式映像

当内模式改变时，只需要修改模式/内模式映像，模式可以不变。

作用：保证**物理独立性**。

# 第二章 关系数据库

## 2.1 数据模型

数据模型是对现实世界数据特征的抽象。

数据模型一般分为三类：

| 类型 | 说明 | 例子 |
|---|---|---|
| 概念模型 | 面向用户和现实世界 | ER 模型 |
| 逻辑模型 | 面向数据库系统 | 关系模型、层次模型、网状模型 |
| 物理模型 | 面向机器存储 | 文件、索引、存储结构 |

---

## 2.2 ER 模型基础

ER 模型是概念模型，常用于数据库设计的概念结构设计阶段。

### 1. 实体 Entity

实体是客观存在并且可以相互区别的事物。

例如：学生、课程、教师、图书。

### 2. 属性 Attribute

属性是实体具有的某种特征。

例如学生实体的属性：学号、姓名、性别、年龄。

### 3. 码 Key

能唯一标识实体的属性或属性组称为码。

例如：学生的学号可以唯一标识学生，所以学号可以作为学生实体的码。

### 4. 联系 Relationship

联系表示实体之间的关联。

例如：学生选修课程，教师讲授课程。

### 5. 联系类型

| 类型 | 含义 | 例子 |
|---|---|---|
| 1:1 | 一个 A 对应一个 B，一个 B 也对应一个 A | 人和身份证 |
| 1:n | 一个 A 对应多个 B，一个 B 只对应一个 A | 班级和学生 |
| m:n | 一个 A 对应多个 B，一个 B 也对应多个 A | 学生和课程 |

---

## 2.3 关系模型基本术语

关系数据库采用二维表表示数据。

| 关系模型术语 | 通俗理解 | 表格中的对应 |
|---|---|---|
| 关系 Relation | 一张表 | Student |
| 元组 Tuple | 一行记录 | 一个学生 |
| 属性 Attribute | 一列 | 学号、姓名 |
| 域 Domain | 属性取值范围 | 性别只能是男/女 |
| 关系模式 | 表结构 | Student(Sno,Sname,Ssex,Sage) |
| 分量 | 元组中的一个属性值 | 某个学生的姓名 |

示例：

```text
Student(Sno, Sname, Ssex, Sage, Sdept)
```

其中 Student 是关系名，Sno、Sname 等是属性。

---

## 2.4 码的概念

### 1. 超码 Super Key

能唯一标识一个元组的属性或属性组。

例如：

```text
{学号}
{学号, 姓名}
{学号, 年龄}
```

只要包含学号，都能唯一标识学生，所以都是超码。

### 2. 候选码 Candidate Key

候选码是**最小的超码**。

关键词：

```text
唯一性 + 最小性
```

例如：如果学号能唯一标识学生，那么 `{学号}` 是候选码。  
但 `{学号, 姓名}` 不是候选码，因为去掉姓名后仍然能唯一标识。

### 3. 主码 Primary Key

从多个候选码中选定一个作为主码。

### 4. 主属性 Prime Attribute

包含在任一候选码中的属性称为主属性。

### 5. 非主属性 Non-prime Attribute

不包含在任何候选码中的属性称为非主属性。

### 6. 外码 Foreign Key

一个关系中的属性引用另一个关系的主码或候选码，则该属性称为外码。

例如：

```text
Student(Sno, Sname, Sdept)
Department(Dno, Dname)
```

如果 `Student.Sdept` 引用 `Department.Dno`，则 `Sdept` 是外码。

---

## 2.5 关系完整性

关系模型有三类完整性约束。

### 1. 实体完整性 Entity Integrity

规则：

```text
主码不能为空，且不能重复。
```

含义：每一行记录必须能被唯一标识。

### 2. 参照完整性 Referential Integrity

规则：

```text
外码的值要么为空，要么等于被参照关系中某个主码值。
```

例如：学生表中的专业号必须在专业表中存在，或者为空。

### 3. 用户定义完整性 User-defined Integrity

用户根据具体业务定义的约束。

例如：

```text
年龄 >= 0
成绩 BETWEEN 0 AND 100
性别 IN ('男', '女')
```

# 第三章 关系代数

关系代数是关系数据库的理论查询语言，常用于考试表达查询过程。

## 3.1 常用关系代数符号

| 操作 | 符号 | 含义 |
|---|---|---|
| 选择 | σ | 选行 |
| 投影 | π | 选列 |
| 并 | ∪ | 合并两个关系的元组 |
| 差 | − | 从一个关系中去掉另一个关系的元组 |
| 笛卡尔积 | × | 两个关系所有元组组合 |
| 连接 | ⋈ | 按条件连接两个关系 |
| 自然连接 | ⋈ | 按同名属性自动等值连接 |
| 除 | ÷ | 表达“全部”“所有”类查询 |

---

## 3.2 选择 Selection

选择是从关系中选出满足条件的元组，即选行。

格式：

```text
σ 条件 (关系名)
```

例：查询年龄大于 20 的学生。

```text
σ Sage>20(Student)
```

---

## 3.3 投影 Projection

投影是从关系中选出若干属性列，即选列。

格式：

```text
π 属性列表 (关系名)
```

例：查询学生的学号和姓名。

```text
π Sno,Sname(Student)
```

注意：投影会自动去掉重复行。

---

## 3.4 选择 + 投影

例：查询计算机系学生的学号和姓名。

```text
π Sno,Sname(σ Sdept='计算机系'(Student))
```

理解顺序：

1. 先从 Student 中选出计算机系学生。
2. 再投影出 Sno 和 Sname。

---

## 3.5 连接 Join

连接用于从多个关系中查询数据。

例：查询学生姓名及其选修课程号。

关系：

```text
Student(Sno, Sname, Sdept)
SC(Sno, Cno, Grade)
```

表达式：

```text
π Sname,Cno(Student ⋈ SC)
```

如果不是自然连接，也可以写成条件连接：

```text
π Sname,Cno(Student ⋈ Student.Sno=SC.Sno SC)
```

---

## 3.6 并、差、交

两个关系做并、差、交时，通常要求它们是并相容的。

并相容要求：

1. 属性个数相同。
2. 对应属性的域相同或相容。

### 1. 并

查询选修了 1 号课程或 2 号课程的学生学号：

```text
π Sno(σ Cno='1'(SC)) ∪ π Sno(σ Cno='2'(SC))
```

### 2. 交

查询既选修 1 号课程又选修 2 号课程的学生学号：

```text
π Sno(σ Cno='1'(SC)) ∩ π Sno(σ Cno='2'(SC))
```

如果不允许用交，可以用差表示：

```text
A ∩ B = A - (A - B)
```

### 3. 差

查询选修了 1 号课程但没有选修 2 号课程的学生学号：

```text
π Sno(σ Cno='1'(SC)) - π Sno(σ Cno='2'(SC))
```

---

## 3.7 除法 Division

除法常用于表达“查询选修了全部课程的学生”这种题。

### 典型题：查询选修了全部课程的学生学号

关系：

```text
SC(Sno, Cno, Grade)
Course(Cno, Cname)
```

表达式：

```text
π Sno,Cno(SC) ÷ π Cno(Course)
```

含义：找出那些在 SC 中与 Course 里的每一个 Cno 都有对应关系的 Sno。

### 做题判断

看到以下关键词，优先考虑除法：

```text
所有
全部
每一个
至少选修了全部……
```

---

## 3.8 关系代数常见题型模板

假设有以下关系：

```text
Student(Sno, Sname, Ssex, Sage, Sdept)
Course(Cno, Cname, Cpno, Ccredit)
SC(Sno, Cno, Grade)
```

### 1. 查询某条件下的学生姓名

题目：查询计算机系学生姓名。

```text
π Sname(σ Sdept='计算机系'(Student))
```

### 2. 查询选修某门课的学生姓名

题目：查询选修 1 号课程的学生姓名。

```text
π Sname(Student ⋈ σ Cno='1'(SC))
```

### 3. 查询成绩大于 90 的学生姓名和课程号

```text
π Sname,Cno(Student ⋈ σ Grade>90(SC))
```

### 4. 查询没有选修 1 号课程的学生学号

```text
π Sno(Student) - π Sno(σ Cno='1'(SC))
```

### 5. 查询既选修 1 号课又选修 2 号课的学生学号

```text
π Sno(σ Cno='1'(SC)) ∩ π Sno(σ Cno='2'(SC))
```

### 6. 查询选修了全部课程的学生学号

```text
π Sno,Cno(SC) ÷ π Cno(Course)
```

# 第四章 SQL 语言

SQL 是结构化查询语言，主要包括：

| 类型 | 说明 | 常见命令 |
|---|---|---|
| DDL | 数据定义语言 | CREATE、ALTER、DROP |
| DML | 数据操纵语言 | SELECT、INSERT、UPDATE、DELETE |
| DCL | 数据控制语言 | GRANT、REVOKE |

---

## 4.1 创建数据库

```sql
CREATE DATABASE TestDb;
```

---

## 4.2 创建表

基本格式：

```sql
CREATE TABLE 表名 (
    列名 数据类型 约束,
    列名 数据类型 约束,
    表级约束
);
```

示例：

```sql
CREATE TABLE Student (
    Sno CHAR(10) PRIMARY KEY,
    Sname VARCHAR(20) NOT NULL,
    Ssex CHAR(2),
    Sage INT CHECK (Sage >= 0),
    Sdept VARCHAR(30)
);
```

---

## 4.3 常见数据类型

| 类型 | 说明 |
|---|---|
| CHAR(n) | 定长字符串 |
| VARCHAR(n) | 变长字符串 |
| INT | 整数 |
| FLOAT | 浮点数 |
| DATE | 日期 |
| DATETIME | 日期时间 |

---

## 4.4 常见约束

| 约束 | 含义 |
|---|---|
| PRIMARY KEY | 主键，非空且唯一 |
| FOREIGN KEY | 外键，保证参照完整性 |
| NOT NULL | 非空 |
| UNIQUE | 唯一 |
| CHECK | 检查约束 |
| DEFAULT | 默认值 |

外键示例：

```sql
CREATE TABLE SC (
    Sno CHAR(10),
    Cno CHAR(10),
    Grade INT CHECK (Grade BETWEEN 0 AND 100),
    PRIMARY KEY (Sno, Cno),
    FOREIGN KEY (Sno) REFERENCES Student(Sno),
    FOREIGN KEY (Cno) REFERENCES Course(Cno)
);
```

---

## 4.6 删除表

```sql
DROP TABLE Student;
```

## 4.7 插入数据

```sql
INSERT INTO Student(Sno, Sname, Ssex, Sage, Sdept)
VALUES ('2024001', '张三', '男', 20, '计算机系');
```

如果给所有列赋值，可以省略列名，但不建议考试中省略：

```sql
INSERT INTO Student
VALUES ('2024001', '张三', '男', 20, '计算机系');
```

---

## 4.8 修改数据

```sql
UPDATE Student
SET Sage = 21
WHERE Sno = '2024001';
```

注意：没有 WHERE 会修改所有记录。

---

## 4.9 删除数据

```sql
DELETE FROM Student
WHERE Sno = '2024001';
```

注意：没有 WHERE 会删除表中所有记录。

---

## 4.10 单表查询

基本格式：

```sql
SELECT 列名
FROM 表名
WHERE 条件;
```

示例：查询计算机系学生姓名。

```sql
SELECT Sname
FROM Student
WHERE Sdept = '计算机系';
```

---

## 4.11 去重 DISTINCT

```sql
SELECT DISTINCT Sdept
FROM Student;
```

---

## 4.12 排序 ORDER BY

升序：

```sql
SELECT Sno, Grade
FROM SC
ORDER BY Grade ASC;
```

降序：

```sql
SELECT Sno, Grade
FROM SC
ORDER BY Grade DESC;
```

默认通常是升序。

---

## 4.13 模糊查询 LIKE

| 写法 | 含义 |
|---|---|
| LIKE '张%' | 以“张”开头 |
| LIKE '%三' | 以“三”结尾 |
| LIKE '%三%' | 包含“三” |
| LIKE '_三' | 两个字符，第二个是“三” |

示例：查询姓张的学生。

```sql
SELECT *
FROM Student
WHERE Sname LIKE '张%';
```

---

## 4.14 范围查询 BETWEEN

```sql
SELECT *
FROM SC
WHERE Grade BETWEEN 80 AND 90;
```

等价于：

```sql
WHERE Grade >= 80 AND Grade <= 90
```

---

## 4.15 集合查询 IN

```sql
SELECT *
FROM Student
WHERE Sdept IN ('计算机系', '软件工程系');
```

---

## 4.16 空值查询 NULL

判断空值不能用 `= NULL`，要用 `IS NULL`。

```sql
SELECT *
FROM Student
WHERE Sdept IS NULL;
```

非空：

```sql
SELECT *
FROM Student
WHERE Sdept IS NOT NULL;
```

---

## 4.17 聚集函数

| 函数 | 含义 |
|---|---|
| COUNT(*) | 统计行数 |
| COUNT(列名) | 统计该列非空值个数 |
| SUM | 求和 |
| AVG | 平均值 |
| MAX | 最大值 |
| MIN | 最小值 |

示例：查询学生总人数。

```sql
SELECT COUNT(*)
```

示例：查询 1 号课程平均成绩。

```sql
SELECT AVG(Grade)
FROM SC
WHERE Cno = '1';
```

---

## 4.18 分组 GROUP BY

题目：查询每门课程的平均成绩。

```sql
SELECT Cno, AVG(Grade)
FROM SC
GROUP BY Cno;
```

---

## 4.19 HAVING

WHERE 和 HAVING 的区别：

| 子句 | 作用对象 |
|---|---|
| WHERE | 分组前，对行筛选 |
| HAVING | 分组后，对组筛选 |

题目：查询平均成绩大于 80 的课程号。

```sql
SELECT Cno, AVG(Grade)
FROM SC
GROUP BY Cno
HAVING AVG(Grade) > 80;
```

## 4.20 多表连接查询

### 1. 等值连接

查询学生姓名和选修课程号。

```sql
SELECT Student.Sname, SC.Cno
FROM Student, SC
WHERE Student.Sno = SC.Sno;
```

### 2. INNER JOIN 写法

```sql
SELECT Student.Sname, SC.Cno
FROM Student
INNER JOIN SC ON Student.Sno = SC.Sno;
```

### 3. 三表连接

查询学生姓名、课程名和成绩。

```sql
SELECT Student.Sname, Course.Cname, SC.Grade
FROM Student, SC, Course
WHERE Student.Sno = SC.Sno
  AND SC.Cno = Course.Cno;
```

## 4.21 嵌套查询

### 1. IN 子查询

查询选修了 1 号课程的学生姓名。

```sql
SELECT Sname
FROM Student
WHERE Sno IN (
    SELECT Sno
    FROM SC
    WHERE Cno = '1'
);
```

### 2. NOT IN 子查询

查询没有选修 1 号课程的学生姓名。

```sql
SELECT Sname
FROM Student
WHERE Sno NOT IN (
    SELECT Sno
    FROM SC
    WHERE Cno = '1'
);
```

### 3. EXISTS 子查询

查询选修了 1 号课程的学生姓名。

```sql
SELECT Sname
FROM Student S
WHERE EXISTS (
    SELECT *
    FROM SC
    WHERE SC.Sno = S.Sno
      AND SC.Cno = '1'
);
```

## 4.22 “全部/所有”类 SQL

题目：查询选修了全部课程的学生学号。

常用双重 NOT EXISTS：

```sql
SELECT Sno
FROM Student S
WHERE NOT EXISTS (
    SELECT *
    FROM Course C
    WHERE NOT EXISTS (
        SELECT *
        FROM SC
        WHERE SC.Sno = S.Sno
          AND SC.Cno = C.Cno
    )
);
```

理解：

```text
不存在这样一门课程：该学生没有选修它。
```

看到“所有、全部、每一门”，SQL 中常用：

```text
NOT EXISTS + NOT EXISTS
```

关系代数中常用：

```text
除法 ÷
```

## 4.23 视图 View

```sql
CREATE VIEW CS_Student AS
```


### 4. 视图的作用

1. 简化用户操作。

# 第五章 数据库安全性与完整性

## 5.1 数据库安全性

数据库安全性是指保护数据库，防止非法使用造成数据泄露、篡改或破坏。

## 5.3 自主存取控制 DAC

## 5.4 授权 GRANT

## 5.6 数据库完整性

数据库完整性是指数据库中数据的正确性、有效性和相容性。

| 类型 | 规则 | SQL 实现 |
|---|---|---|
| 实体完整性 | 主码非空且唯一 | PRIMARY KEY |
| 参照完整性 | 外码必须引用已有主码或为空 | FOREIGN KEY |

## 5.7 完整性约束 SQL 示例

```sql
CREATE TABLE Student (
```

```sql
CREATE TABLE Course (
    Cno CHAR(10) PRIMARY KEY,

);
```

选课表：

```sql
CREATE TABLE SC (
    Sno CHAR(10),

);
```

# 第六章 关系数据库理论

这一章是考试难点，重点是：

```text
函数依赖 → 候选码 → 范式判断 → 模式分解
```

---

## 6.1 函数依赖 Functional Dependency

设关系模式 R(U)，X 和 Y 是属性集。  
如果对于 R 的任意两个元组，只要 X 上的值相同，Y 上的值也一定相同，则称：

```text
X 函数决定 Y
```

记作：

```text
X → Y
```

例：

```text
学号 → 姓名
学号 → 性别
学号 → 专业
```

因为一个学号只能对应一个学生。

---

## 6.2 平凡函数依赖与非平凡函数依赖

### 1. 平凡函数依赖

如果 Y 是 X 的子集，则 X → Y 是平凡函数依赖。

例：

```text
(Sno, Cno) → Sno
```

因为 Sno 本来就在左边。

### 2. 非平凡函数依赖

如果 Y 不是 X 的子集，则 X → Y 是非平凡函数依赖。

例：

```text
Sno → Sname
```

---

## 6.3 完全函数依赖与部分函数依赖

设 X → Y，且 X 是属性组。

### 1. 完全函数依赖

如果 Y 依赖于整个 X，而不依赖于 X 的任何真子集，则称 Y 完全函数依赖于 X。

记作：

```text
X -F→ Y
```

例：

```text
(Sno, Cno) → Grade
```

成绩由学生和课程共同决定，只看 Sno 或只看 Cno 都不行。

### 2. 部分函数依赖

如果 Y 依赖于 X，但也依赖于 X 的某个真子集，则称 Y 部分函数依赖于 X。

例：

```text
(Sno, Cno) → Sname
Sno → Sname
```

因为 Sname 只由 Sno 决定，不需要 Cno，所以 Sname 对 (Sno, Cno) 是部分依赖。

---

## 6.4 传递函数依赖

如果：

```text
X → Y
Y → Z
Y 不决定 X
Z 不属于 Y
```

则称：

```text
Z 传递依赖于 X
```

例：

```text
Sno → Sdept
Sdept → Mname
```

则：

```text
Sno → Mname
```

Mname 传递依赖于 Sno。

---

## 6.5 候选码求法

候选码是能推出全部属性的最小属性集。

### 基本方法：属性闭包

属性集 X 的闭包记为：

```text
X+
```

表示由 X 根据函数依赖能推出的所有属性。

### 求闭包步骤

1. 先把 X 自己放入 X+。
2. 查看函数依赖集 F 中有没有左部包含在 X+ 里的依赖。
3. 如果有，就把右部属性加入 X+。
4. 重复直到 X+ 不再变化。
5. 如果 X+ 包含关系模式全部属性，则 X 是超码。
6. 如果 X 是最小超码，则 X 是候选码。

### 示例

关系模式：

```text
R(A, B, C, D)
F = { A → B, B → C, C → D }
```

求 A+：

```text
A+ = {A}
A → B，所以加入 B：A+ = {A, B}
B → C，所以加入 C：A+ = {A, B, C}
C → D，所以加入 D：A+ = {A, B, C, D}
```

所以 A 能推出全部属性，A 是候选码。

---

## 6.6 范式 Normal Form

范式用于减少数据冗余和异常。

常见异常：

| 异常 | 含义 |
|---|---|
| 插入异常 | 想插入某些信息，但因为缺少其他信息插不进去 |
| 删除异常 | 删除某条记录时，把不该删除的信息也删了 |
| 修改异常 | 同一信息重复存储，修改时容易不一致 |

---

## 6.7 第一范式 1NF

如果关系中每个属性都是不可再分的数据项，则属于 1NF。

简单理解：

```text
表中每个单元格只能放一个值，不能再拆。
```

不满足 1NF 的例子：

```text
学生(学号, 姓名, 电话号码集合)
```

因为电话号码集合可以再拆。

---

## 6.8 第二范式 2NF

在 1NF 基础上，消除非主属性对候选码的部分函数依赖。

关键词：

```text
2NF：消除部分依赖
```

常见发生场景：

```text
候选码是组合码
```

例：

```text
SC(Sno, Cno, Sname, Cname, Grade)
候选码：(Sno, Cno)
函数依赖：
(Sno, Cno) → Grade
Sno → Sname
Cno → Cname
```

Sname 只依赖 Sno，Cname 只依赖 Cno，都是对组合码的部分依赖。  
所以不满足 2NF。

分解：

```text
Student(Sno, Sname)
Course(Cno, Cname)
SC(Sno, Cno, Grade)
```

---

## 6.9 第三范式 3NF

在 2NF 基础上，消除非主属性对候选码的传递函数依赖。

关键词：

```text
3NF：消除传递依赖
```

例：

```text
Student(Sno, Sname, Sdept, Mname)
函数依赖：
Sno → Sname, Sdept
Sdept → Mname
```

因为：

```text
Sno → Sdept → Mname
```

Mname 传递依赖于 Sno，所以不满足 3NF。

分解：

```text
Student(Sno, Sname, Sdept)
Dept(Sdept, Mname)
```

---

## 6.10 BCNF

BCNF 比 3NF 更严格。

定义：

```text
对于每一个非平凡函数依赖 X → Y，X 都必须是超码。
```

判断口诀：

```text
左边不是超码，就不满足 BCNF。
```

3NF 和 BCNF 区别：

| 范式 | 要求 |
|---|---|
| 3NF | 允许某些右边是主属性的特殊情况 |
| BCNF | 每个决定因素都必须是超码 |

一般考试中：

```text
BCNF 一定满足 3NF，但 3NF 不一定满足 BCNF。
```

---

## 6.11 范式判断快速方法

判断一个关系属于第几范式，可以按这个顺序：

1. 看属性是否原子：不原子则不是 1NF。
2. 求候选码。
3. 找主属性和非主属性。
4. 看非主属性是否部分依赖候选码：有则不是 2NF。
5. 看非主属性是否传递依赖候选码：有则不是 3NF。
6. 看每个函数依赖左边是否都是超码：不是则不满足 BCNF。

---

## 6.12 模式分解

模式分解是把一个关系模式分解成多个关系模式，以减少冗余和异常。

分解要关注两个性质：

```text
无损连接性
保持函数依赖
```

### 1. 无损连接分解

分解后再自然连接，能恢复原关系，不产生错误数据。

对于二元分解：

```text
R 分解为 R1 和 R2
```

如果满足：

```text
R1 ∩ R2 → R1 - R2
或者
R1 ∩ R2 → R2 - R1
```

则分解具有无损连接性。

简单理解：

```text
公共属性必须能决定其中一个关系的剩余属性。
```

### 2. 保持函数依赖

分解后，各子关系上的函数依赖合起来，仍能推出原来的函数依赖集。

注意：

```text
有些分解无损连接，但不保持函数依赖。
有些分解保持函数依赖，但不一定达到 BCNF。
```

# 第七章 数据库设计

数据库设计是考试重点，尤其是 ER 图和 ER 图转关系模式。

## 7.1 数据库设计步骤

记忆：

```text
需 概 逻 物 实 运
```

## 7.2 需求分析

目标：弄清楚用户需要管理哪些数据、实现哪些功能。

```text
需求说明书、数据字典、数据流图等
```

## 7.3 概念结构设计

概念结构设计通常使用 ER 图。

目标：把现实世界抽象成实体、属性和联系。

产物：

```text
ER 图
```

---

## 7.4 逻辑结构设计

目标：把 ER 图转换为某种 DBMS 支持的数据模型，通常是关系模型。

产物：

```text
关系模式
```

```text
Student(Sno, Sname, Ssex, Sage, Sdept)
```

## 7.5 物理结构设计

3. 确定数据存放位置。
4. 考虑查询效率。

---

## 7.6 ER 图画法

ER 图基本元素：

| 元素 | 常见表示 |
|---|---|
| 实体 | 矩形 |
| 属性 | 椭圆 |
| 联系 | 菱形 |
| 实体与属性连接 | 线 |
| 实体与联系连接 | 线，并标注 1、n、m |

### ER 图做题步骤

1. 从题干中找名词，判断实体。
2. 从题干中找描述信息，判断属性。
3. 找实体之间的动词，判断联系。
4. 判断联系类型：1:1、1:n、m:n。
5. 给每个实体确定主码。
6. 如果联系有自己的属性，也要标出来。

---

## 7.7 ER 图转关系模式规则

### 1. 实体转换

每个实体转换成一个关系模式。

实体属性变成关系属性，实体码变成关系主码。

例：

```text
学生实体：学生(学号, 姓名, 性别, 年龄)
转换为：Student(Sno, Sname, Ssex, Sage)
主码：Sno
```

---

### 2. 1:1 联系转换

方法一：单独转换成一个关系模式。

方法二：把任一方的主码加入另一方作为外码，同时加入联系属性。

例：

```text
人(Person) 和 身份证(IDCard) 是 1:1 联系
```

可以设计为：

```text
Person(Pid, Pname, CardNo)
IDCard(CardNo, Address)
```

其中 `Person.CardNo` 是外码。

---

### 3. 1:n 联系转换

通常把 1 端的主码加入 n 端作为外码，并把联系属性也加入 n 端。

例：

```text
班级 Class 和 学生 Student 是 1:n
```

转换为：

```text
Class(Cno, Cname)
Student(Sno, Sname, Cno)
```

其中 `Student.Cno` 是外码。

---

### 4. m:n 联系转换

m:n 联系必须单独转换成一个关系模式。

该关系模式包含两端实体的主码，并把它们共同作为主码，同时加入联系属性。

例：

```text
学生 Student 和 课程 Course 是 m:n 联系：选修
选修有属性 Grade
```

转换为：

```text
Student(Sno, Sname)
Course(Cno, Cname)
SC(Sno, Cno, Grade)
```

其中：

```text
SC 的主码：(Sno, Cno)
Sno 是外码，引用 Student(Sno)
Cno 是外码，引用 Course(Cno)
```

---

### 5. 多值属性转换

多值属性通常单独转换成一个关系模式。

例：学生有多个电话号码。

```text
Student(Sno, Sname)
StudentPhone(Sno, Phone)
```

其中：

```text
StudentPhone 的主码可为 (Sno, Phone)
Sno 是外码
```

---

### 6. 弱实体转换

弱实体转换成关系模式时，要把所依赖强实体的主码加入进来，通常与弱实体的部分码共同组成主码。

---

## 7.8 数据库设计题答题模板

### 题目要求：根据描述画 ER 图

答题步骤：

```text
1. 确定实体：学生、课程、教师……
2. 确定属性：学生有学号、姓名、性别……
3. 确定主码：学生的主码为学号……
4. 确定联系：学生选修课程，教师讲授课程……
5. 确定联系类型：学生与课程是 m:n，教师与课程是 1:n……
6. 如果联系有属性：选修联系有成绩……
```

### 题目要求：ER 图转关系模式

答题格式：

```text
学生(学号, 姓名, 性别, 年龄)
主码：学号

课程(课程号, 课程名, 学分)
主码：课程号

选修(学号, 课程号, 成绩)
主码：(学号, 课程号)
外码：学号 引用 学生(学号)，课程号 引用 课程(课程号)
```

# 第八章 数据库恢复技术

恢复技术用于保证数据库在故障后能够恢复到正确状态。

---

## 8.1 事务 Transaction

事务是用户定义的一个数据库操作序列，这些操作要么全做，要么全不做。

例：银行转账。

```text
A 账户扣 100
B 账户加 100
```

这两个操作必须作为一个整体执行。

---

## 8.2 事务的 ACID 特性

| 特性 | 含义 |
|---|---|
| 原子性 Atomicity | 事务中的操作要么全做，要么全不做 |
| 一致性 Consistency | 事务执行前后数据库都必须处于一致状态 |
| 隔离性 Isolation | 并发事务之间不能互相干扰 |
| 持久性 Durability | 事务一旦提交，对数据库的改变就是永久的 |

记忆：

```text
ACID = 原子性 + 一致性 + 隔离性 + 持久性
```

---

## 8.3 事务的状态

常见事务状态：

```text
活动状态 → 部分提交状态 → 提交状态
活动状态 → 失败状态 → 中止状态
```

---

## 8.4 故障类型

### 1. 事务故障

某个事务由于程序错误、输入错误等原因不能继续执行。

恢复方法：撤销该事务，即 UNDO。

### 2. 系统故障

系统停止运行，例如断电、操作系统崩溃，但磁盘数据没有破坏。

恢复方法：

```text
UNDO 未提交事务
REDO 已提交事务
```

### 3. 介质故障

磁盘损坏等导致数据库文件破坏。

恢复方法：使用备份副本和日志恢复。

---

## 8.5 日志文件

日志文件用于记录事务对数据库的更新操作。

常见日志记录：

```text
<T start>
<T, X, old_value, new_value>
<T commit>
<T abort>
```

作用：

1. 事务故障时用于 UNDO。
2. 系统故障时用于 UNDO/REDO。
3. 介质故障时配合备份恢复。

---

## 8.6 先写日志原则 WAL

WAL：Write-Ahead Logging，先写日志原则。

核心要求：

```text
在把数据修改写入数据库之前，必须先把对应日志记录写入稳定存储。
```

这样故障后才能根据日志恢复。

---

## 8.7 UNDO 和 REDO

### 1. UNDO 撤销

把未完成事务对数据库的修改撤销掉。

使用日志中的旧值 old_value。

### 2. REDO 重做

把已经提交事务的操作重新执行一遍。

使用日志中的新值 new_value。

---

## 8.8 检查点 Checkpoint

检查点用于减少恢复时需要扫描的日志范围。

检查点会把内存中的脏数据写入磁盘，并在日志中记录检查点信息。

作用：

```text
提高恢复效率，减少恢复时间。
```

---

## 8.9 恢复策略总结

| 故障类型 | 恢复策略 |
|---|---|
| 事务故障 | UNDO 该事务 |
| 系统故障 | UNDO 未提交事务，REDO 已提交事务 |
| 介质故障 | 装入备份，利用日志 REDO |

---

# 第九章 并发控制

并发控制用于保证多个事务同时执行时，数据库仍然保持正确。

---

## 9.1 并发操作带来的问题

### 1. 丢失修改 Lost Update

两个事务同时修改同一数据，后一个修改覆盖前一个修改。

### 2. 读脏数据 Dirty Read

一个事务读到了另一个事务尚未提交的数据。  
如果另一个事务后来回滚，则前一个事务读到的数据就是错误的。

### 3. 不可重复读 Non-repeatable Read

一个事务两次读取同一数据，中间另一个事务修改并提交，导致两次读取结果不同。

### 4. 幻读 Phantom Read

一个事务按条件查询多行记录，另一个事务插入或删除符合条件的记录，导致再次查询时结果集行数变化。

---

## 9.2 调度 Schedule

调度是多个事务操作的执行顺序。

### 1. 串行调度

多个事务一个接一个执行，没有交叉。

一定正确，但效率低。

### 2. 并发调度

多个事务的操作交叉执行。

效率高，但可能产生不一致。

---

## 9.3 可串行化 Serializability

如果一个并发调度的执行结果与某个串行调度结果相同，则该调度是可串行化的。

```text
可串行化调度是正确调度。
```

---

## 9.4 冲突可串行化

两个操作冲突需要满足：

1. 属于不同事务。
2. 操作同一个数据项。
3. 至少有一个是写操作。

冲突类型：

```text
读-写冲突：R(X) 与 W(X)
写-读冲突：W(X) 与 R(X)
写-写冲突：W(X) 与 W(X)
```

读-读不冲突。

---

## 9.5 优先图判断冲突可串行化

判断步骤：

1. 每个事务作为一个结点。
2. 如果 Ti 的某个操作与 Tj 的某个操作冲突，并且 Ti 的操作先于 Tj，则画边 Ti → Tj。
3. 如果优先图无环，则调度冲突可串行化。
4. 如果有环，则不是冲突可串行化。

结论：

```text
优先图无环 ⇔ 冲突可串行化
```

---

## 9.6 封锁 Locking

封锁是在事务访问数据前先对数据加锁。

### 1. 排他锁 X 锁

也叫写锁。

如果事务 T 对数据 A 加了 X 锁，则 T 可以读和写 A，其他事务不能再对 A 加任何锁。

### 2. 共享锁 S 锁

也叫读锁。

如果事务 T 对数据 A 加了 S 锁，则 T 可以读 A，但不能写 A。其他事务可以继续对 A 加 S 锁，但不能加 X 锁。

---

## 9.7 锁的相容矩阵

| 已加锁 / 请求锁 | S 锁 | X 锁 |
|---|---|---|
| S 锁 | 相容 | 不相容 |
| X 锁 | 不相容 | 不相容 |

记忆：

```text
只有 S 和 S 相容，其余都不相容。
```

---

## 9.8 两段锁协议 2PL

两段锁协议规定事务分为两个阶段：

### 1. 扩展阶段

事务可以申请锁，但不能释放锁。

### 2. 收缩阶段

事务可以释放锁，但不能再申请新锁。

结论：

```text
遵守两段锁协议的调度一定是可串行化的。
```

注意：

```text
两段锁协议可能产生死锁。
```

---

## 9.9 活锁和死锁

### 1. 活锁

某个事务一直等待，但系统一直在运行，事务长期得不到执行机会。

解决方法：先来先服务。

### 2. 死锁

两个或多个事务互相等待对方释放资源，导致都无法继续执行。

例：

```text
T1 等 T2 释放 B
T2 等 T1 释放 A
```

解决方法：

1. 死锁预防。
2. 死锁检测。
3. 死锁解除，通常撤销某个事务。

---

## 9.10 隔离级别

常见 SQL 隔离级别：

| 隔离级别 | 可能出现的问题 |
|---|---|
| Read Uncommitted | 脏读、不可重复读、幻读 |
| Read Committed | 不可重复读、幻读 |
| Repeatable Read | 幻读 |
| Serializable | 基本避免上述问题 |

隔离级别越高，并发程度通常越低。

# 第十章 高频简答题模板

## 10.1 数据库系统的特点

答：数据库系统的主要特点包括：数据结构化，数据共享性高、冗余度低，数据独立性高，数据由 DBMS 统一管理和控制。DBMS 可以提供安全性控制、完整性控制、并发控制和恢复功能。

---

## 10.2 三级模式两级映像

答：数据库系统的三级模式包括外模式、模式和内模式。外模式是用户看到的局部逻辑结构，模式是数据库的全局逻辑结构，内模式是数据的物理存储结构。两级映像包括外模式/模式映像和模式/内模式映像，前者保证逻辑独立性，后者保证物理独立性。

---

## 10.3 关系模型的三类完整性

答：关系模型的完整性包括实体完整性、参照完整性和用户定义完整性。实体完整性要求主码不能为空且不能重复；参照完整性要求外码要么为空，要么引用被参照关系中已经存在的主码；用户定义完整性是用户根据实际业务定义的约束，如年龄范围、成绩范围等。

---

## 10.4 事务的 ACID 特性

答：事务具有原子性、一致性、隔离性和持久性。原子性指事务中的操作要么全做，要么全不做；一致性指事务执行前后数据库都保持一致状态；隔离性指并发事务之间不能互相干扰；持久性指事务一旦提交，对数据库的修改就是永久的。

---

## 10.5 什么是可串行化调度

答：如果一个并发调度的执行结果与某个串行调度的执行结果相同，则称该调度是可串行化调度。可串行化调度是并发事务正确性的标准。判断冲突可串行化时，可以构造优先图，如果优先图无环，则该调度是冲突可串行化的。

---

## 10.6 什么是两段锁协议

答：两段锁协议规定事务的执行分为扩展阶段和收缩阶段。在扩展阶段，事务只能申请锁，不能释放锁；在收缩阶段，事务只能释放锁，不能再申请新锁。遵守两段锁协议的调度一定是可串行化的，但可能产生死锁。

---

## 10.7 数据库设计的步骤

答：数据库设计一般包括需求分析、概念结构设计、逻辑结构设计、物理结构设计、数据库实施、数据库运行和维护六个阶段。其中概念结构设计通常用 ER 图表示，逻辑结构设计通常把 ER 图转换为关系模式。

---

# 第十一章 易错判断点

1. 数据库系统不仅包括数据库和 DBMS，还包括应用程序、用户、DBA 等。
2. 一个数据库只有一个模式，但可以有多个外模式。
3. 外模式/模式映像保证逻辑独立性，模式/内模式映像保证物理独立性。
4. 主码不能为空，这是实体完整性要求。
5. 外码可以为空，也可以引用另一个表中已经存在的主码。
6. 投影操作会去掉重复元组。
7. 选择是选行，投影是选列。
8. 关系代数中的除法常用于“全部、所有、每一个”类查询。
9. WHERE 是分组前筛选，HAVING 是分组后筛选。
10. 判断 NULL 不能用 `= NULL`，要用 `IS NULL`。
11. 视图是虚表，通常只保存定义，不直接保存数据。
12. 2NF 消除部分函数依赖，3NF 消除传递函数依赖。
13. BCNF 要求每个非平凡函数依赖的左部都是超码。
14. BCNF 一定满足 3NF，但 3NF 不一定满足 BCNF。
15. 无损连接分解不一定保持函数依赖。
16. 保持函数依赖的分解不一定是 BCNF。
17. 事务故障通常 UNDO，系统故障通常 UNDO 未提交事务并 REDO 已提交事务。
18. WAL 原则要求先写日志，再写数据库。
19. 只有 S 锁和 S 锁相容，X 锁与任何锁都不相容。
20. 遵守两段锁协议的调度一定可串行化，但可能死锁。
21. 读-读不冲突，读-写、写-读、写-写冲突。
22. 优先图无环，则调度冲突可串行化；有环则不是冲突可串行化。

# 第十二章 考前速记版

## 12.1 绪论速记

```text
DB：数据库，数据集合
DBMS：数据库管理系统，管理数据库的软件
DBS：数据库系统，DB + DBMS + 应用程序 + DBA + 用户
DBA：数据库管理员
```

三级模式：

```text
外模式：用户视图
模式：全局逻辑结构
内模式：物理存储结构
```

两级映像：

```text
外模式/模式映像：逻辑独立性
模式/内模式映像：物理独立性
```

---

## 12.2 关系数据库速记

```text
关系 = 表
元组 = 行
属性 = 列
域 = 取值范围
候选码 = 唯一性 + 最小性
主码 = 选定的候选码
外码 = 引用其他表主码的属性
```

完整性：

```text
实体完整性：主码非空且唯一
参照完整性：外码为空或引用已有主码
用户定义完整性：CHECK、NOT NULL 等
```

---

## 12.3 关系代数速记

```text
σ：选择，选行
π：投影，选列
⋈：连接
∪：并
-：差
÷：除，用于全部/所有
```

---

## 12.4 SQL 速记

```sql
SELECT 列
FROM 表
WHERE 行条件
GROUP BY 分组列
HAVING 组条件
ORDER BY 排序列;
```

执行理解顺序：

```text
FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
```

---

## 12.5 范式速记

```text
1NF：属性不可再分
2NF：消除非主属性对候选码的部分依赖
3NF：消除非主属性对候选码的传递依赖
BCNF：每个决定因素都是超码
```

---

## 12.6 数据库设计速记

设计步骤：

```text
需求分析 → 概念结构设计 → 逻辑结构设计 → 物理结构设计 → 数据库实施 → 运行维护
```

ER 图：

```text
实体：矩形
属性：椭圆
联系：菱形
```

ER 转关系：

```text
实体 → 关系模式
1:n → 1 端主码加入 n 端
m:n → 单独建关系，双方主码共同作为主码
多值属性 → 单独建关系
```

---

## 12.7 恢复技术速记

```text
事务 ACID：原子性、一致性、隔离性、持久性
事务故障：UNDO
系统故障：UNDO 未提交 + REDO 已提交
介质故障：备份 + 日志恢复
WAL：先写日志，再写数据库
```

---

## 12.8 并发控制速记

```text
并发问题：丢失修改、脏读、不可重复读、幻读
S 锁：读锁，共享锁
X 锁：写锁，排他锁
只有 S-S 相容
2PL：先加锁后解锁，解锁后不能再加锁
2PL 保证可串行化，但可能死锁
优先图无环：冲突可串行化
```
